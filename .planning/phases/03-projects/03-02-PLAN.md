<objective>
Make the build resilient: if the GitHub API call fails at build time (network error, 4xx/5xx, malformed JSON), the build must still succeed by falling back to a committed `src/data/projects.json` file. The user-facing page renders cards either way; the fallback is invisible to visitors. A single intentional `console.warn` at build time announces the fallback so the developer notices in CI logs.

Purpose: Plan 01 left the build dependent on network availability and GitHub API uptime — a transient API blip would break deploys. This plan implements decision D-06 (build must always succeed; stale cached data is preferable to a broken build) by adding a try/catch + JSON fallback. After this plan ships, `next build` succeeds even when the API is unreachable.

Output: One new file (src/data/projects.json) and one modified file (src/lib/projects.ts with try/catch + fallback path + documented warn).
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Committed JSON file → build pipeline | The fallback JSON is committed to git. A malicious commit could inject arbitrary repo names/descriptions/URLs into the rendered page. |
| Build host → GitHub REST API | Same as Plan 01; unchanged. The try/catch does not change the fetch posture, only the failure handling. |
| Build log → CI surface | A single intentional console.warn appears in build logs. Must not contain secrets. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-05 | Tampering | src/data/projects.json (committed fallback) | mitigate | The file is committed via normal git review (the developer reviews the diff before merging). The same XSS protections apply at render time as for live API data: ProjectCard renders all text via JSX text-children (auto-escaped) and uses href attributes only. JSON shape is validated by the Plan 01 + Plan 02 acceptance criteria (round-trip with Project type) before merge. Risk reduced to "an attacker who can land a PR can change the fallback content" — same as for any other source file. |
| T-03-06 | Information Disclosure | src/lib/projects.ts catch block (build log warn) | mitigate | The warn message format is `[projects] GitHub fetch failed, using src/data/projects.json fallback: <error.message>`. error.message originates from native fetch (network error strings) or from the explicit `Error('GitHub API ${res.status}')` thrown in the try block — neither path includes the GITHUB_TOKEN value. The Authorization header value is never serialized to the error message. Acceptance criteria check that the warn line uses `error.message` (not `error.stack` or full request dumps). Build logs in GitHub Actions are visible to repo collaborators only, so even if a leak were to occur it would be contained — but the design eliminates the leak path entirely. The W-01 automated fallback test deliberately uses an obviously-invalid placeholder token (`ghp_invalidtoken_force_401_for_fallback_test_…`) to avoid leaking any real token value into CI logs. |
| T-03-07 | Tampering | console.warn ESLint compliance | accept | Project ESLint rules (next/core-web-vitals + project conventions) do not currently flag console.warn (only console.log per typescript hooks.md). The eslint-disable-next-line comment is added defensively in case a future stricter rule lands. No actual lint suppression of a real-today rule is happening. |
| T-03-SC | Tampering | npm installs | accept | No new packages installed in this plan. Only one new committed file (projects.json) and one modified TS file (projects.ts). No package.json modification. |
