<threat_model>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Actions runner -> GitHub Pages | Runner builds the static site and uploads an artifact; deploy-pages publishes it. Token-scoped to the workflow run. |
| GitHub Actions runner -> GitHub REST API | Build-time `fetchProjects()` calls api.github.com using `GITHUB_TOKEN` (1,000 req/hr authenticated rate limit). |
| Untrusted contributor (pull request) -> deploy job | `on: push: branches: [main]` means PRs do NOT trigger deploys. Only commits already on `main` deploy. Branch protection (out of scope here) further hardens this. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Tampering | Action versions in deploy.yml | accept | Pin to majors (`@v4`, `@v5`, `@v3`) per CLAUDE.md mandate. SHA-pinning offered no benefit for a personal-site Pages workflow and has high maintenance cost (every Dependabot bump becomes a full SHA review). The blast radius is a single static site rebuild — accepted risk. |
| T-04-02 | Information Disclosure | GITHUB_TOKEN in build logs | mitigate | GitHub Actions automatically masks `secrets.*` values in logs. The token is passed only via `env:` on the build step, never echoed. Verified by visual log inspection in Task 2 checkpoint. |
| T-04-03 | Tampering | Pages source enabled by manual UI step | mitigate | Task 2 checkpoint requires the human to confirm Source is set to "GitHub Actions" (NOT a deploy-from-branch fallback). A misconfigured Source would silently publish whatever is on the `gh-pages` branch instead of the workflow output. |
| T-04-04 | Denial of Service | Concurrent workflow runs racing the Pages deployment | mitigate | `concurrency: group: pages, cancel-in-progress: false` queues runs serially. Never cancels an in-flight deploy (avoids leaving Pages in a partial state). |
| T-04-05 | Elevation of Privilege | Workflow over-broad permissions | mitigate | Workflow-level `permissions:` block declares EXACTLY `contents: read`, `pages: write`, `id-token: write`. No `actions: write` or `repo: admin` granted. id-token is required for OIDC-based deploy-pages@v5; pages: write is required to publish. |
| T-04-SC | Tampering | npm/pip/cargo installs | mitigate | `npm ci` uses package-lock.json — no new packages introduced in this plan. Existing lockfile is the supply-chain audit trail. No checkpoint needed because no new packages are added in Phase 4. |

</threat_model>
