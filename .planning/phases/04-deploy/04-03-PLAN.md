<interfaces>
<!-- Reference: GitHub's official Next.js Pages starter workflow caches .next/cache like this. -->
<!-- This is the canonical pattern — do not invent a different cache key shape. -->

actions/cache@v4 inputs we will use:
- `path:` — exactly `.next/cache` (Next.js 16 uses this for the build output cache)
- `key:` — `${{ runner.os }}-nextjs-${{ hashFiles('package-lock.json') }}-${{ hashFiles('src/**', 'next.config.ts', 'postcss.config.mjs', 'tsconfig.json') }}`
  - Three components:
    1. OS (so a Linux cache never restores on macOS).
    2. Lockfile hash — scoped to repo-root `package-lock.json` only (NOT `**/package-lock.json`, which would also hash any nested lockfiles, including any in node_modules).
    3. Source-file hash — scoped to `src/**` plus the build-relevant config files (`next.config.ts`, `postcss.config.mjs`, `tsconfig.json`). This explicitly excludes `node_modules/**`, `out/**`, `.next/**`, `.planning/**`, and any other non-source trees.
- `restore-keys:` — fallback prefix `${{ runner.os }}-nextjs-${{ hashFiles('package-lock.json') }}-` so a partial-source change still gets a warm cache to start from

Why scoped globs (NOT `**/*.ts`):
- The unscoped form `**/*.ts` would hash files inside `node_modules/` if it existed at hash time. Even when `node_modules/` is `.gitignore`d, `actions/cache@v4` runs after `npm ci`, so node_modules IS present on disk during hashFiles evaluation. An unscoped glob would produce a cache key that varies with dependency-tree minor patches even when our own source has not changed — defeating the cache.

Insertion order (between Plan 01 build job steps):
1. checkout@v4
2. setup-node@v4
3. configure-pages@v5
4. npm ci
5. **NEW: actions/cache@v4 (this plan)**
6. npm run build
7. upload-pages-artifact@v3

Why between npm ci and npm run build:
- After `npm ci` we know node_modules is in place (cache restore should not interfere)
- Before `npm run build` so .next/cache is restored before next build runs
- Putting it before `npm ci` would cache it but the build would not see the restored copy if subsequent steps mutated the directory layout

<threat_model>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Actions cache store -> runner filesystem | Cached `.next/cache` is restored into the runner workspace. The cache contents are scoped per-branch and per-key, isolated from other repos and other branches. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-09 | Tampering | Stale cache producing incorrect build output | mitigate | The cache key includes a hash of `src/**` plus build-relevant config files, plus the lockfile hash. Any source change invalidates the source-component of the key. The lockfile hash component invalidates on any dependency change. The combination guarantees that a cache hit only occurs when both deps and source are byte-identical to a previous build — in which case the cached `.next/cache` IS the correct artifact for that input. The scoped glob (excluding node_modules) prevents accidental over-invalidation from dependency-tree noise. |
| T-04-10 | Information Disclosure | .next/cache leaking secrets | accept | Next.js `.next/cache` is build-time output (incremental compile artifacts, optimized images). It does NOT contain secret values — secrets are read from `process.env` at build time and inlined into JS bundles only when explicitly referenced in the code. The Phase 3 build-time `GITHUB_TOKEN` is used to call api.github.com but is never persisted into the build output (verified in Phase 3 Plan 03 SUMMARY: `grep -rE "GITHUB_TOKEN\|Bearer " out/_next/static/` returns 0 matches). The cache contents are at most a subset of the build output, so the same property holds. |
| T-04-11 | Denial of Service | Cache corruption blocking builds | mitigate | If cache restoration fails or produces a broken build, the build step fails and the workflow goes red. The next push or cron run will use a different key (any source change) or hit the same key but `actions/cache@v4` will retry. The fallback is to manually delete the cache via `gh cache delete` or to push a no-op commit to bump the source hash. Because this plan is non-blocking, a cache-related red can also be resolved by reverting just the cache step without affecting Plans 01/02. |
| T-04-SC | Tampering | npm/pip/cargo installs | mitigate | No new packages introduced. `actions/cache@v4` is a GitHub-published action pinned per CLAUDE.md. |

</threat_model>
